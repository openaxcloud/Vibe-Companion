PANEL 1 : SHELL / TERMINAL

Statut : ❌ CASSÉ
Symptôme : "Shell Disconnected" + 6x "Connection error: server error"
Le WebSocket shell ne se connecte jamais au workspace
Solutions :

Vérifier le endpoint WebSocket /ws/shell ou /ws/terminal dans server/index.ts — s'assurer qu'il est bien monté et que le serveur WebSocket accepte les connexions
Vérifier que le processus PTY (pseudo-terminal) se lance correctement côté serveur (node-pty ou equivalent)
Vérifier les logs serveur au moment de la connexion shell — probable erreur d'init du container/workspace
Ajouter un retry avec backoff exponentiel côté frontend au lieu de 6 retries instantanés
Afficher un message d'erreur utile : "Le terminal ne peut pas se connecter. Vérifiez que le serveur est en cours d'exécution." au lieu de "server error"




PANEL 2 : CODE EDITOR

Statut : ⚠️ Bloqué sur "Loading editor..."
L'éditeur CodeMirror/Monaco ne se charge jamais
Solutions :

Vérifier que le chunk JavaScript de l'éditeur se charge correctement (pas de 404 sur les assets)
Vérifier la console pour des erreurs de module lazy-loading — possible timeout réseau
L'éditeur dépend probablement de la connexion au workspace/runtime. Si le shell est cassé, l'éditeur peut aussi être bloqué
Ajouter un timeout avec fallback : si l'éditeur ne charge pas en 10s, afficher "Erreur de chargement — Réessayer" avec un bouton
Vérifier que le fichier sélectionné (index.html) est bien récupéré via l'API /api/projects/:id/files/:path




PANEL 3 : PREVIEW

Statut : ⚠️ Bloqué sur "Building your app... Planning project structure..."
Le preview ne rend jamais le contenu
Solutions :

Le preview dépend du runtime/serveur de dev. Si le shell est cassé, le preview ne peut pas builder
Vérifier le endpoint de preview : le serveur doit compiler le projet et retourner le HTML
Alternative : charger le index.html du projet directement dans un iframe via srcdoc au lieu d'attendre un build complet
Ajouter un timeout : si le build prend plus de 30s, afficher "Le build a échoué" avec les logs d'erreur
Ajouter un bouton "Rafraîchir le preview" pour forcer un re-build




PANEL 4 : CONSOLE

Statut : ⚠️ "Runtime Connecting..." — ne complète jamais
Solutions :

Même root cause que le Shell — la connexion au runtime est cassée
Quand le shell sera fixé, la console devrait fonctionner aussi
Ajouter un indicateur de statut clair : "Connecté" (vert) / "Déconnecté" (rouge) / "En cours..." (orange)
Afficher les logs serveur dans la console même sans runtime actif (logs statiques du dernier build)




PANEL 5 : AGENT CHAT (sidebar gauche)

Statut : ⚠️ Partiel — "Reconnecting... (1/10)"
Le chat AI se connecte puis perd la connexion WebSocket
Solutions :

Vérifier le WebSocket /ws/agent ou /ws/chat — même problème que le shell
Le message "Reconnecting 1/10" montre que le retry existe mais échoue à chaque fois
Vérifier l'authentification WebSocket — le token de session est peut-être expiré à cause du bug sameSite
Ajouter un fallback HTTP polling si le WebSocket échoue (SSE ou long-polling)
Quand déconnecté, griser l'input et afficher "Chat temporairement indisponible" au lieu de laisser taper sans réponse




PANEL 6 : ACTIONS TAB

Statut : ❌ CASSÉ — "Empty error thrown while loading module: AgentActionsPanel"
Le lazy-loading du composant échoue après 3 tentatives
Solutions :

Vérifier que le chunk AgentActionsPanel existe dans dist/assets/ — possible qu'il n'ait pas été buildé
npm run build et vérifier que tous les chunks sont générés sans erreur
Le composant fait probablement un import dynamique (React.lazy) — vérifier le path d'import
Ajouter un error boundary avec un bouton "Réessayer" autour du lazy component
Vérifier si le composant dépend d'une API qui est down — l'erreur "empty error" suggère une exception non catchée
