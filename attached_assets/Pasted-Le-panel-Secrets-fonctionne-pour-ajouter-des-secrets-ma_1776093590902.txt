Le panel Secrets fonctionne pour ajouter des secrets manuellement, mais 3 améliorations sont nécessaires :
1. Injection des secrets dans le runtime — Quand le projet tourne (Run), les secrets du projet doivent être injectés comme variables d'environnement dans le process enfant. Vérifie dans server/terminal.ts et server/localWorkspaceManager.ts que quand on spawne un process pour le projet, les secrets sont récupérés depuis la DB (SELECT * FROM env_vars WHERE projectId = ?), déchiffrés (AES-256-GCM), et passés dans process.env du child process. Le code existe déjà dans le terminal handler (decryptEnvVars) — assure-toi que le localWorkspaceManager fait la même chose quand il lance npm run dev.
2. L'agent IA doit proposer les secrets manquants — Quand l'agent génère du code qui utilise une API key (ex: process.env.STRIPE_SECRET_KEY), l'agent doit :

Détecter les process.env.XXX dans le code qu'il génère
Vérifier si le secret XXX existe dans les secrets du projet
Si non, envoyer un message au chat : 'Your project needs STRIPE_SECRET_KEY. Please add it in the Secrets panel.'
Optionnellement, ouvrir automatiquement le panel Secrets avec le nom pré-rempli

Implémente ça dans le système prompt de l'agent : ajoute une instruction 'After creating files that use environment variables, check if those variables are configured. If not, inform the user which secrets they need to add.'
3. Distinction App Secrets vs Account Secrets :

App Secrets = secrets spécifiques à CE projet (ex: DATABASE_URL de cette app)
Account Secrets = secrets partagés entre TOUS les projets de l'utilisateur (ex: ANTHROPIC_API_KEY)
Les deux onglets existent déjà dans l'UI. Vérifie que :
Les App Secrets sont stockés avec projectId en DB
Les Account Secrets sont stockés avec userId sans projectId
Au runtime, les deux sont injectés (Account Secrets + App Secrets, avec App qui override Account si même nom)
L'agent IA ne peut PAS lire les valeurs des secrets (sécurité) mais peut lire les noms pour vérifier s'ils existent

Corrige et rebuild."
