🔴 CRITIQUES (10 points) :
1. .env.production traqué dans Git
	∙	Fichier : .env.production (confirmé avec git ls-files)
	∙	Contenu exposé : sk_live_ Stripe, DATABASE_URL avec mdp Neon, OPENAI_API_KEY, GEMINI_API_KEY, JWT_SECRET, SESSION_SECRET, ENCRYPTION_KEY, REDIS_URL + password, ADMIN_PASSWORD en clair (AdminECode2025$Secure!), NEON_API_KEY, SENTRY_DSN, FIGMA_API_KEY, EXPO_TOKEN, TAVILY_API_KEY, DOKPLOY_API_KEY, XAI_API_KEY, MOONSHOT_API_KEY
	∙	Impact : compromission totale — transactions Stripe frauduleuses, accès DB complet, usage API à tes frais
	∙	Fix : retirer du git history (BFG Repo-Cleaner), rotation de TOUTES les clés, ajouter au .gitignore
2. Double session middleware
	∙	Fichiers : server/index.ts (lignes 89-106) → MemoryStore + server/routes.ts (lignes 1083-1112) → connectPgSimple (PostgreSQL)
	∙	Express applique le dernier enregistré. Si routes.ts échoue au chargement → fallback MemoryStore → toutes les sessions perdues au restart
	∙	Impact : 1000 users connectés → restart → tout le monde déconnecté
	∙	Fix : supprimer MemoryStore de index.ts, garder uniquement PostgreSQL
3. 6 configurations sameSite contradictoires
	∙	server/index.ts:103 → "none"
	∙	server/routes.ts:1102 → "none" en prod / "lax" en dev
	∙	server/auth/session-manager.ts:157 → "strict"
	∙	server/middleware/passport-setup.ts:44 → "lax"
	∙	server/utils/security.ts:649 → "lax"
	∙	server/middleware/security.ts:321 → "strict"
	∙	Impact : CSRF possible avec sameSite: "none", cookies perdus sur redirections avec "strict"
	∙	Fix : unifier sur sameSite: "lax" + secure: true partout
4. routes.ts monolithique — 21,420 lignes, 578 endpoints
	∙	Fichier : server/routes.ts
	∙	Impact : un seul crash dans un handler fait tomber l’ensemble du serveur. Impossible à maintenir, tester ou débuguer
	∙	Fix : découper en modules (auth/, projects/, files/, ai/, admin/, payments/)
5. 495+ blocs catch {} vides
	∙	Disséminés dans tout le codebase serveur
	∙	Impact : erreurs silencieuses en production — tu ne sauras jamais ce qui plante. Debugging impossible
	∙	Fix : logger toutes les erreurs, au minimum console.error(e) + Sentry
6. Rate limiting en mémoire uniquement
	∙	Fichier : middleware de rate limiting utilise une Map en mémoire
	∙	Impact : reset à chaque redémarrage. Avec plusieurs workers/instances, chaque instance a ses propres limites → bypass facile (10x si 10 instances). Memory leak car la Map n’est jamais nettoyée
	∙	Fix : migrer vers Redis-backed rate limiting
7. Tool executor sans vérification de propriété du projet
	∙	Fichier : server/agent/tool-definitions.ts + executeur
	∙	L’agent AI exécute des opérations fichier (create_file, etc.) sans vérifier que le projet appartient à l’utilisateur authentifié
	∙	Impact : un utilisateur pourrait potentiellement manipuler des fichiers d’un autre projet
	∙	Fix : ajouter vérification project.userId === session.userId avant chaque opération
8. Source maps exposées en production
	∙	Fichier : vite.config.ts — sourcemap: true
	∙	Les fichiers .map sont inclus dans dist/ et servis publiquement
	∙	Impact : n’importe qui peut reconstruire ton code source frontend complet
	∙	Fix : sourcemap: false ou sourcemap: 'hidden' en production
9. Bootstrap timer global avec memory leak
	∙	Fichier : store autonomousBuildStore
	∙	Timer state global avec race condition. Chaque refresh/navigation accumule des timers non nettoyés
	∙	Impact : fuite mémoire progressive, comportement imprévisible après longue utilisation
	∙	Fix : cleanup des timers dans useEffect return / store destroy
10. Race conditions webhooks Stripe
	∙	Fichier : service Stripe dans server/
	∙	Pas de garde d’idempotence sur le traitement des webhooks. Un même événement Stripe peut être traité 2 fois
	∙	Impact : double facturation possible, états de paiement incohérents
	∙	Fix : stocker les <event.id> traités et vérifier avant traitement
