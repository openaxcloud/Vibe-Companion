ÉLEVÉS (8 points) :
11. Risque XSS dans le rendu markdown
	∙	Fichier : RichMessageContent component
	∙	Rend du HTML depuis les réponses AI sans sanitization complète
	∙	Impact : si l’AI génère du HTML malicieux (ou si un prompt inject du JS), exécution de code dans le navigateur de l’utilisateur
	∙	Fix : utiliser DOMPurify pour sanitizer tout le HTML rendu
12. Stale closures dans AgentConversationStore
	∙	Fichier : src/stores/agentConversationStore.ts
	∙	Le state management Zustand avec persist middleware capture des références périmées de messages
	∙	Impact : messages perdus, écrasés par des anciennes références lors d’updates rapides
	∙	Fix : utiliser getState() dans les callbacks au lieu de capturer la closure
13. Stripe fallback vers clés de test
	∙	Variables d’environnement Stripe tombent en fallback sur des valeurs test si les variables prod sont manquantes
	∙	Impact : en cas de mauvaise config env, les paiements partent vers Stripe test → transactions perdues sans erreur visible
	∙	Fix : fail hard si STRIPE_SECRET_KEY n’est pas défini en prod
14. Pas de rate limiting sur les uploads de fichiers
	∙	Les endpoints d’upload n’ont pas de limite de débit
	∙	Impact : un attaquant peut uploader massivement et remplir le disque / consommer la bande passante
	∙	Fix : ajouter rate limit + taille max par fichier et par user
15. Conflit session lifetime — 24h vs 7 jours
	∙	passport-setup.ts:43 → maxAge: 24h
	∙	routes.ts:1102 → maxAge: 7 jours
	∙	index.ts:102 → maxAge: 7 jours
	∙	Impact : comportement imprévisible, users déconnectés aléatoirement
	∙	Fix : unifier sur 24h (plus sécurisé) ou 7 jours (meilleur UX) — un seul endroit
16. Endpoints auth dupliqués
	∙	/api/auth/login défini dans index.ts (ligne 212) ET dans routes.ts (ligne ~335)
	∙	/api/register et /api/auth/register aussi dupliqués
	∙	Impact : comportement différent selon quel handler s’exécute (MemoryStore vs PG, CSRF token ou pas)
	∙	Fix : supprimer les doublons de index.ts, garder uniquement routes.ts
17. N+1 queries sur l’export users
	∙	L’endpoint export fait 1 + N requêtes DB (101 queries pour un user avec 100 projets)
	∙	Impact : avec 1000 users × 100 projets = 100,000 requêtes DB → timeout / crash
	∙	Fix : utiliser des JOINs ou eager loading
18. Index DB manquants
	∙	Pas d’index composite sur (projectId, createdAt) pour les requêtes analytics
	∙	Pas d’index sur les foreign keys courantes (projects.userId, files.projectId)
	∙	Impact : full table scans → requêtes lentes sous charge
	∙	Fix : ajouter les index avec une migration Drizzle
